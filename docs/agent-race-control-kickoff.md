# Agent Race Control ‚Äî Kickoff Brief

You are helping me build a small, focused desktop app from scratch. Read this whole brief first, then **restate the plan and your assumptions back to me and wait for my confirmation before writing any code.** Work in phases; pause for my sign-off at the end of each phase.

## What we're building

**Agent Race Control** (package/repo slug: `agent-race-control`) is a session-aware terminal cockpit for Claude Code on **native Windows**.

The problem it solves is concrete: I run several Claude Code sessions at once, each in its own terminal window ‚Äî plus more terminals for `npm run dev`, builds, and the like ‚Äî and I waste time hunting the Windows taskbar for the right one. Agent Race Control consolidates **all of it** into **one window, one taskbar icon**: every Claude Code session *and* the plain shells I run alongside them. An overview I can glance at and a terminal I can drive, together. If a terminal is open, it lives in Agent Race Control.

## Non-negotiable ethos: minimal

Minimalism is a hard requirement, not a preference. I evaluated the existing tools (Kanban Code, Nimbalyst, claude-code-kanban, and others) and rejected them as bloated. The bloat was never the embedded terminals ‚Äî it was everything bolted around them. **Agent Race Control is Windows Terminal, except each tab knows whether it's a Claude Code session ‚Äî and shows its name, color, and status ‚Äî or a plain shell. Nothing more.**

### Explicitly OUT of scope (do not build these, and flag me if you're tempted)
- No git worktree or branch *management* ‚Äî I have git. (Read-only **display** of the current branch/worktree is in scope ‚Äî pure observability, like status-from-hooks; see "Branch/worktree tree" below. We never create, switch, or delete anything.)
- No PR / GitHub integration.
- No kanban columns or drag-and-drop.
- No task-dependency graphs.
- No theme gallery ‚Äî one clean look. (A **light / dark / system** appearance toggle is in scope; a gallery of decorative themes is not.)
- No database ‚Äî small JSON files for local state, nothing heavier.

Every "wouldn't it be nice" gets checked against this list. When in doubt, leave it out and ask.

## Non-negotiable ethos: full-fidelity Claude Code

Agent Race Control must run the **unmodified `claude` CLI in a real pseudo-terminal** ‚Äî it does not wrap, proxy, embed via SDK/headless mode, or reimplement Claude Code. This is the single most important design constraint, and it's *why* Agent Race Control exists: a real PTY session is indistinguishable from launching `claude` in Windows Terminal, so **every feature the terminal has is present by construction** ‚Äî remote control, rewind/checkpoints, `--resume`, slash commands (including `/btw` side questions and their overlay keys), **background agents and the agent view** (left-arrow detach/switch, `/attach`, `claude agents` ‚Äî discovery is file-based under `~/.claude/jobs/` + a per-user supervisor daemon, so it's launch-method-agnostic), plan mode, hooks, MCP, mouse and keyboard interaction, true color, hyperlinks ‚Äî all of it, 100% identical.

Other tools (Kanban Code, the Claude desktop app, GitKraken Kepler, etc.) integrate Claude Code through the SDK, headless mode, or a non-TTY wrapper, so those interactive features are missing or degraded. Agent Race Control deliberately does **not** do that. The burden on us is therefore inverted: our job is to **not interfere**. Concretely:

- Spawn the real `claude` binary in a real PTY (ConPTY via `node-pty`); never intercept, rewrite, or filter the byte stream in either direction.
- Pass **all keystrokes** straight through to the PTY ‚Äî including chord/escape sequences Agent Race Control might otherwise want (e.g. Esc-Esc for rewind, Ctrl+C, Ctrl+R). App-level shortcuts must not shadow anything Claude Code uses.
- **Zero new muscle memory.** Inside the terminal, Agent Race Control invents no shortcuts of its own ‚Äî it follows stock Windows Terminal conventions (e.g. Ctrl+Shift+C/V and right-click for copy/paste; Ctrl+C is Claude's interrupt, untouched; **Ctrl+= / Ctrl+- / Ctrl+0 zoom the window** ‚Äî the same keys Windows Terminal and VS Code bind, consciously shadowing the rare readline undo chord). Moving between Agent Race Control and a regular terminal must require no change in habits, in either direction.
- Pass through the full environment, a correct `TERM` (`xterm-256color`), accurate rows/cols on every resize, clipboard copy/paste, and OSC-8 hyperlinks. One deliberate carve-out: strip the env vars an enclosing Claude Code session injects into its children (`CLAUDECODE`, `CLAUDE_CODE_SESSION_ID`, ‚Ä¶) ‚Äî fidelity means matching a *fresh terminal*, not the launcher's ancestry, and leaked markers make spawned claudes act as nested child sessions (which silently disables transcript persistence ‚Äî no resume). Found the hard way.
- Dropping a file on the terminal pastes its quoted path at the cursor ‚Äî the same bytes Windows Terminal sends, and how images/files get handed to Claude Code in a terminal. (The window itself never navigates: `will-navigate` is blocked.)
- Never steer Claude Code's appearance: no `COLORFGBG`, theme flags, or config injection. Agent Race Control's light/dark toggle themes Agent Race Control's chrome only (see *Appearance*). Claude Code picks its own theme.
- If a feature works in Windows Terminal but not in Agent Race Control, that's a bug in Agent Race Control ‚Äî not a feature we chose to omit.

## Layout

- **Timing tower (left):** a list of every session showing its name, a color dot, and a status indicator. Named after the F1 broadcast graphic it copies ‚Äî the left-edge column of drivers with position and status; "tower" in code and shorthand.
- **Right pane:** one terminal, showing whichever session I clicked in the tower.
- The tower is the overview; the pane is the cockpit. Same window. Glance to find, click to drive.
- Visual reference: **the F1 timing tower** ‚Äî a column of colored entries with names and live status, the whole race readable at a glance; click one to go on board.
- **Status dot colors (settled 2026-07-12, replacing CI-badge semantics):** traffic lights read from the **user's** point of view ‚Äî every color answers "is this session mine to act on?" Claude sessions: **running = red** (the agent is driving, hands off), **waiting = amber, pulsing** (it's asking for you), **idle = green** (your turn ‚Äî a fresh session starts here, at its prompt). **A live shell is neutral full-strength ink (`fg`)** ‚Äî a power LED, not an agent state: "running" only means its PTY is alive (we can't and won't inspect the foreground process), and green stays exclusive to "a Claude awaits you". **exited = faded muted gray** for both types ‚Äî a shell's dot literally just dims when it dies. All hexes are existing Primer tokens (`danger`/`attention`/`success`/`fg`) with roles remapped ‚Äî zero new colors. **Light-theme refinement (2026-07-14):** the dots use the brighter Primer *.emphasis* mid-tones on light (`success` #2da44e, `attention` #bf8700) instead of the *.fg* text tones (#1a7f37/#9a6700), which read muddy as small filled dots on white; `danger` is unchanged (.fg == .emphasis == #cf222e) and dark keeps its already-bright .fg tones. Accepted trade-offs: a tower of busy agents glows red (correct under this framing ‚Äî all stop lights, nothing for you), and red is spent on "busy", so a hypothetical bad-exit state can never claim it (fine ‚Äî don't over-model status). **Status RGB setting (2026-07-15):** an opt-in toggle (Settings menu, **off by default**) that recolors the three agent-status dots to pure traffic-light RGB ‚Äî running `#ff0000`, waiting `#ffaa00`, idle `#00ff00` ‚Äî identical in both themes. The Primer-token defaults above are unchanged; this is a single named switch for people who want literal traffic-light hues, not per-color customization (the shell power-LED, exited-gray, and the close/clear hovers all keep reading the palette tokens). Persisted as `statusRgb` in the state JSON. Scoped in code to dedicated `--dot-running/waiting/idle` CSS vars fed by the `dots` derived in `App.svelte`.
- **TODO marker (issue #3, 2026-07-17):** a cosmetic per-session "revisit later" flag overlaid on the status dot, **both session types**. The dot becomes a small `<button>` (mirrors the close-button pattern ‚Äî native button, `stopPropagation`); **left-click toggles TODO, toggle-only ‚Äî it does not focus the row** (a dedicated control; click elsewhere on the row to focus). While on, the dot pulses blue regardless of the underlying status color (reuses the existing `pulse` keyframe), riding the same Status RGB toggle as the status dots: the **theme accent** (`palette.chrome.accent` ‚Äî #0969da light / #2f81f7 dark) by default, **pure `#0000FF`** when Status RGB is on (matching the pure-RGB traffic lights). Fed via a `--dot-todo` var off the `dots` derived. It **auto-clears the next time the underlying status changes color**: every status write (hooks `applyStatus`, the keystroke nudge, exit) funnels through one `setStatus(session, next)` that drops TODO on a value change ‚Äî each status is a distinct color, so a value change *is* a color change; a *running* session flagged TODO therefore clears the moment it finishes (running‚Üíidle). Clicking again clears it manually and the dot returns to its live status color (`session.status` is tracked underneath the whole time, so there's no "previous state" to restore). Persisted as `todo` per session in the state JSON (additive/optional) so it **survives restart** ‚Äî set directly on restore, *not* via `setStatus`, so a relaunch's status defaults don't count as the clearing change. **Ethos:** cosmetic only ‚Äî no effect on sorting, focus, spawn, resume, or any logic. It reuses the existing dot (no new UI element) and carries no columns/workflow-states/ordering, so the "no kanban" line holds ‚Äî a sticky note, not a board.

One bordered control, not floating glyphs.

### Repo cards (issue #5 ‚Äî grouping settled 2026-07-17, card presentation settled 2026-07-19)
Every group renders as a **card**: a faint **wash** of its colour (8% over the tower bg) with a 2px colour **tab welded to the card's left edge**, laid out on a **12-column grid** so markers, titles, dots, icons and names align to fixed tracks. A **git repo** card nests **repo ‚Üí branch ‚Üí sessions** (the one blessed exception to the otherwise-flat "no nesting" rule ‚Äî running parallel agents across a repo's worktrees is the app's core use case); a non-git folder card is just title ‚Üí sessions. This is pure observation (see the qualified out-of-scope line above ‚Äî we display the branch, never manage it). The card presentation **supersedes** the earlier F1-stripe/tree look (edge stripe + a 1px repo-colour trunk + 12px branch tabs), chosen over ~20 playground iterations; the *grouping* model is unchanged.
- **12-column grid (`repeat(12, 1fr)` per row ‚Äî tracks align across rows without subgrid, since every row is the same width):** col 1 = the repo **title** (marker `none-left`: the repo name is a flush-left heading, no icon) + the branch icon; col 2 = branch names + session dots (both flush-left ‚Üí one clean vertical edge); col 3 = session type icon; col 4+ = session name. **Default card spec (playground-tuned, kept in memory):** wash 8% ¬∑ edge tab 2px ¬∑ no border ¬∑ radius 2px ¬∑ marker none-left ¬∑ icon 12px ¬∑ fonts title/branch/session = 14/12/12.
- **Grouping = Model B:** the top group is the **repo**, keyed on its *shared* git dir so **every worktree of one repo groups under one card**. `git rev-parse --path-format=absolute --git-common-dir` returns the same absolute `<repo>/.git` from every worktree (verified empirically against a real linked worktree, git 2.54); `dirname` ‚Üí the repo root, basename ‚Üí the repo name. Each branch row is a **cwd**; its branch (`--abbrev-ref HEAD`) labels the row, with `‚Üí <worktree-basename>` shown when the worktree dir differs from the repo name. **Detached HEAD** (`--abbrev-ref` ‚Üí `HEAD`) shows a short SHA (`--short HEAD`, a second call ‚Äî `--abbrev-ref` is a sticky rev-parse mode).
- **Git info is the app's first (and only) git subprocess.** `src/main/git.ts` ‚Äî `getGitInfo(cwd)` via `execFile('git', ‚Ä¶, { timeout })`, **fail-open**: any error/timeout ‚Üí `{ isRepo: false }`, never a throw or hang. Exposed as `window.arc.git.info(cwd)`; cached per cwd in the renderer (`gitInfo`, not persisted ‚Äî recomputed on load and on **window focus** so a `git checkout` in a shell updates the card; no `.git/HEAD` fs-watch). A non-git cwd (or info not yet landed) renders as a plain card until it resolves.
- **Ordering reuses `dirOrder`** (a list of cwds ‚Äî no schema bump): cards are *derived* by clustering cwds by repo in first-appearance order (`groupCwds`, unit-tested). DnD ‚Äî the **whole card** is the group's drag handle, moving its cwd block (`moveGroup`); **sessions** reorder within their cwd (their row stops `dragstart` from bubbling so a row reorder can't become a card move). No branch-level DnD.
- **Drop feedback (settled 2026-07-19):** a group drop previews as a **2px accent insertion line floating in the margin gap** where the card will land ‚Äî above the hovered card (*before*) or below it (*after*), chosen by which half of the card the pointer is over ‚Äî instead of highlighting the whole target. It's a `::after` in the gap, so the card is `overflow: visible` and the colour tab rounds its own left corners to stay clipped. "After a card" resolves to "before the next **visible** card": the DOM is the source of visible order (hidden groups aren't rendered), so the hovered card's next `.card` sibling is it, and `null` ‚Üí append ‚Äî which is how a card reaches the very top or bottom. Mid-drag the card's inner content drops `pointer-events` (keyed off `data-dragging` on the tower body) so native DnD's per-child `dragenter`/`dragleave` and `:hover` can't flicker the line or light up card chrome as the cursor crosses the target.
- **Colour = the repo's primary (first-in-order) worktree's colour** ‚Äî drives the card wash + edge tab (no `dirColors` re-keying). Right-click the card title to recolor; `setGroupColor` writes *every* worktree so it stays stable as the primary changes.
- **Collapse (settled 2026-07-19):** clicking a card's **name** folds it to just its title (branch + session rows hidden) and unfolds it ‚Äî a *click*, not a drag, so dragging the card to reorder and right-click-to-recolour both still work on the same title. A collapsed card leads its title with a **roll-up status dot** so a folded card still reads at a glance, most-urgent first: **waiting** (a session wants you) ‚Ä∫ a running **Claude** (agent driving) ‚Ä∫ any other **running** (a live shell) ‚Ä∫ **idle** ‚Ä∫ **exited** ‚Äî it carries the winning session's type, so a lone running shell reads neutral (`plain`), never red, the same per-type colours and waiting-pulse the rows use. Collapsed state is **per group key**, persisted as `collapsed` (a list of keys ‚Äî additive/optional, filtered to alive groups on save, mirroring `dirColors`); default expanded. This ships the previously-deferred collapse + roll-up; the once-floated *roll-up-on-an-expanded-title* is dropped ‚Äî expanded cards show their real per-session dots.

### Two session types
- **Claude session** ‚Äî bash that `exec`s straight into `claude` (see spawn spec below). The PTY *is* the Claude process: when claude exits, the session is exited. Full status set.
- **Shell session** ‚Äî a plain interactive Git Bash, for `npm run dev`, builds, git, or anything else. This is what makes Agent Race Control the *only* terminal window I need ‚Äî without it I'd still have stray terminals on the taskbar, defeating the point. Status is `running`/`exited` only.
- Both types live in the same tower, same look: name, color dot, status. New-session UI offers exactly these two choices. A shell session is not a degraded Claude session ‚Äî it's a first-class terminal.

### Claude Code stays pure ‚Äî Agent Race Control never steers its theme
- Agent Race Control does **not** hint, force, or sync Claude Code's internal theme. No `COLORFGBG`, no config injection, no theme flags. Claude Code renders exactly however it would in Windows Terminal.
- This is a direct consequence of the full-fidelity ethos: Agent Race Control sets a genuine terminal palette (the GitHub colors above) and stops there. If Claude Code auto-detects the real terminal background and adjusts, that's *Claude* reading the true environment ‚Äî which is correct and pure. Agent Race Control's appearance toggle and Claude Code's own theme are independent by design, and that's fine even if they don't match.

## Stack (decided ‚Äî don't re-litigate unless you hit a real blocker)

- **Electron + TypeScript.**
- **Terminal UI:** `@xterm/xterm` (note the `@xterm` scope ‚Äî the bare `xterm` package is legacy) plus `@xterm/addon-fit` for resize and `@xterm/addon-clipboard` for **OSC 52** ‚Äî Claude Code copies to the system clipboard via OSC 52 (e.g. `/btw`'s `c` key), and xterm.js only honors it with this addon. (Verify addon compat with xterm 6 in Phase 2.)
- **OSC 8 hyperlinks (fixed 2026-07-18):** clicking a link Claude Code prints (e.g. a login URL) approved the browser's `confirm()` but then silently did nothing. Root cause: xterm's default link activation opens a *blank* popup first (`window.open()` with no URL, a popup-blocker dodge) and only sets `.location.href` on it afterward ‚Äî but our `setWindowOpenHandler` (main/index.ts) sees that blank URL, fails the http(s) check, and denies it, so the popup is `null` and the real URL is never navigated to. Fix: a custom `linkHandler.activate` on the `Terminal` (Terminal.svelte) that keeps the same confirm-before-opening UX but calls `window.open(uri, '_blank')` with the real URL directly, so it flows through the existing http(s)-only guard correctly. Same guard, same escape hatch as the preview's links ‚Äî just no longer routed through the broken two-step popup.
- **PTY backend:** `node-pty` (a full pseudo-terminal is required because Claude Code is a full-screen TUI ‚Äî plain `child_process` won't do).
- **Tower / renderer UI:** **Svelte** (with TypeScript). Chosen deliberately: Svelte compiles to small vanilla JS with ~no runtime, so it keeps the dependency/footprint story consistent with the minimal ethos while giving us real reactivity for the tower and appearance state. No React.

### Technical specifics that are easy to get wrong ‚Äî get these right
- `node-pty` is a **native module** ‚Äî but since v1.1 it's **Node-API** and ships **prebuilt Windows binaries** (`prebuilds/win32-x64/`: `pty.node`, `conpty.node`, `winpty.dll`, `winpty-agent.exe`). Node-API is ABI-stable across Node *and* Electron, so **no `@electron/rebuild` step is needed** ‚Äî the old "rebuild for Electron's ABI" advice is obsolete for this package. (Verified the hard way: the 1.1.0 npm tarball can't even be gyp-rebuilt on Windows ‚Äî it ships without winpty's `GetCommitHash.bat`. Prebuilds are the only supported path from the tarball.) Caveat: the loader prefers `build/Release` over `prebuilds/`, so never leave a stale/partial `build/` dir inside `node_modules/node-pty`. On Windows it uses ConPTY under the hood, which renders Claude Code's TUI correctly.
- **Security model:** `contextIsolation: true`, `nodeIntegration: false`. Run `node-pty` in the **main process**; expose a minimal, explicit API to the renderer via a **preload `contextBridge`** (e.g. `spawn`, `write`, `onData`, `resize`, `kill`). Shuttle bytes over IPC ‚Äî keystrokes renderer‚Üímain, output main‚Üírenderer, plus resize events.
- **Shell (decided, non-negotiable):** spawn **Git Bash** (`bash.exe`) via node-pty and run `claude` inside it ‚Äî Git Bash is Claude Code's recommended shell on Windows. Every terminal Agent Race Control opens is a Git Bash session; we do not offer PowerShell/cmd/WSL as alternatives.
- **Bash resolution (decided):** adopt VS Code's battle-tested algorithm (`terminalProfiles.ts`) essentially verbatim. **Never PATH-resolve `bash.exe`** ‚Äî on Windows that can hit the WSL stub (`WindowsApps\bash.exe`) or legacy WSL launcher (`System32\bash.exe`). Instead:
  1. PATH-resolve **`git.exe`** (no WSL doppelg√§nger, and it's the Git the user actually uses) and walk up from `<root>\cmd\git.exe` to the install root.
  2. Add standard install roots from env vars (`ProgramW6432`, `ProgramFiles`, `ProgramFiles(X86)`, `%LOCALAPPDATA%\Programs`) and the scoop locations (`scoop\apps\git\current`, `scoop\apps\git-with-openssh\current`).
  3. For each root, try `bin\bash.exe` first (the wrapper that sets up the MSYS env ‚Äî never `git-bash.exe`, which spawns its own mintty window), then `usr\bin\bash.exe`. **First path that exists wins.**
  4. No registry lookup ‚Äî keeps deps at zero and matches VS Code.
  5. If nothing is found, fail loudly with a clear "Git Bash not found" error. Optional `bashPath` override key in the state JSON as an escape hatch (no UI for it).
- **Spawn spec (decided):** always a **login interactive** shell ‚Äî sources the user's profile, so PATH (e.g. `~/.local/bin` where `claude` lives) resolves byte-for-byte as in a regular Git Bash terminal; same args VS Code uses for its Git Bash profile.
  - **Shell session:** `bash.exe --login -i`
  - **Claude session:** `bash.exe --login -i -c 'exec claude --session-id <uuid> --settings <Agent Race Control-hooks-file>'` (resume: `--resume <uuid>` instead of `--session-id`). The `--login` still sources the profile before `-c` runs, so `claude` resolves from the user's real PATH; `exec` makes bash *become* claude, so the PTY's lifetime **is** the claude process's lifetime ‚Äî `running`/`exited` fall straight out of the PTY with no child-process tracking. Deliberate consequence: when claude exits, the session ends rather than dropping to a prompt ‚Äî the tower models Claude sessions, not shells; if I want a shell, I open a shell session.
  - The two extra flags are the *only* deviation from a bare `claude` launch, and both are user-typeable CLI arguments, not config mutation: `--session-id` pins the id Agent Race Control already needs for resume, and `--settings` adds the observability-only status hooks (see open items). Claude's behavior is otherwise byte-identical.

- **Status ‚Äî keep it simple.** Because Agent Race Control *spawns* the sessions, it owns each PTY, so `running` / `exited` are known directly from the process. For the `waiting-for-input` signal, wire a lightweight Claude Code **hook** (Notification / Stop) that writes a small marker Agent Race Control watches. - **Name & color (naming model revised 2026-07-12):** a Claude session's display name *is* the conversation's live name, read from the terminal title. Renaming in the tower types `/rename <new name>` into the session (documented inline-argument form), and a rename made inside Claude Code flows back through the title ‚Äî **two-way by construction, one source of truth (Claude's)**. One wrinkle (observed): Claude doesn't re-emit the terminal title immediately on `/rename`, so the tower optimistically sets the *title itself* to the new name, overwritten by the next real title emission ‚Äî `name` never holds Claude labels (it's the shell-only label field; the earlier name-override scheme got wiped by the session's first title event). A shell rename is a local label that shadows the title (Git Bash titles the path). The cwd basename is never shown as a session name: the old spawn default (`name` = basename) is gone, and pre-v2 state files are discarded wholesale on load (see the pre-1.0 no-compat policy under *Stack ‚Üí State*), so a deliberate shell name is never second-guessed. **Blessed injections (`/color`, `/rename`) are typed only at the idle prompt** (tightened in review 2026-07-13, one shared `injectCommand` guard): while `waiting` the trailing Enter would answer the open permission/question dialog, and while `running` the input box may hold a draft; a rename that can't inject changes nothing (a visible no-op beats a silent revert), and blur-committing an unchanged rename injects nothing. Known asymmetry (verified empirically; upstream: claude-code#41466/#49293): `/color` set *inside* the TUI can't be read back (lands in no file, no escape sequence), so sync is one-way app‚Üísession. Persist shell names and directory colors in the state JSON.
- **Persistence across restarts:** the PTY is a child of Agent Race Control, so closing Agent Race Control ends the process ‚Äî but the Claude Code *session* persists on disk. On relaunch, reopen a session with `claude --resume <session-id>`. This gives tmux-like continuity **without tmux** (I'm deliberately off WSL/tmux).

- **Hooks are blind to user interrupts (found + fixed 2026-07-12):** documented behavior ‚Äî `Stop` fires only on normally-completed turns, and *no* hook fires when a permission/question dialog is dismissed (Esc) or a running turn is interrupted, so dots stuck amber/red after a cancel. Same lone `0x1b` byte, indistinguishable, and no hook fires either way ‚Äî so we drop the running-interrupt nudge (Ctrl+C still interrupts-to-idle instantly). Accepted trade-off: a genuine Esc-interrupt of a running turn no longer greens until your next action ‚Äî the *safe* wrong direction (false-red "busy" beats false-green "your turn" while Claude is still working). Optimistic nudge only ‚Äî the next hook event stays authoritative. ~~Parked: `PermissionRequest` and `Notification` payload types~~ **verified empirically same day** (probe: throwaway `claude --settings` session POSTing every event to a capture server) **and fixed:** (1) the false amber was real ‚Äî exactly 60s after `Stop`, `Notification` fires with `notification_type: "idle_prompt"` ("Claude is waiting for your input"); the status server now **drops that type** (all other notification types still map to `waiting` ‚Äî conservative default for unknown future types). Deliberately *not* mapped to `idle`: unverified whether the nag can also fire while a dialog sits open, and `Stop` already covers the green. - **Spinner-driven idle ‚Äî the interrupt watchdog (issue #6 follow-up, 2026-07-17):** an empirical probe (node-pty, four cases: normal completion / Esc-interrupt / `/btw`+Esc / Ctrl+C, watching hooks + title + transcript side by side) confirmed what the docs say ‚Äî **no hook fires on any user interrupt or menu/dialog dismissal** (design: hooks are turn-lifecycle, not UI events). ## Next phase (post-v1, agreed 2026-07-12): read-only conversation preview

A toggleable **markdown preview of the conversation** as a second view of the session. Read-only observation; the PTY byte stream stays untouched, so the fidelity ethos holds by construction.

- **Data source:** the transcript JSONL at `~/.claude/projects/<cwd-encoded>/<uuid>.jsonl` ‚Äî the path is fully known because we pin `--session-id` at spawn (encoding: every non-alphanumeric dashed, see *Session files* above; `transcriptPath()` in `transcript.ts` is the single owner, and `pty.ts` reports the *effective* spawn cwd back so a dead-path fallback can't desync the tail from where Claude actually writes). **Main process tails the file** (fs.watch + incremental reads of appended lines), parses entries, pushes them to the renderer over IPC (same pattern as the status server, file-flavored) ‚Äî **in bounded batches with yields between them** (added in review 2026-07-13): the first read of a watch is the whole file, and main's event loop also pumps every PTY, so it must never block for a full parse nor ship one unbounded IPC array. **fs.watch is backstopped by a ~1s stat poll while armed (added 2026-07-15):** Windows `fs.watch` on the directory *drops* change events ‚Äî notably the final append of a turn, the byte range carrying the last assistant message ‚Äî and the watch has no retry for a missed event, so flipping Terminal‚ÜîPreview mid-turn could strand the preview one turn behind (the ~50ms mount catch-up read races the final flush; if the following write's event is dropped, those bytes are never re-read). The poll ‚Äî one stat/sec, only while a preview is open, watch still the low-latency path ‚Äî schedules a read whenever the file has grown past the read offset, so the final state always lands within ~1s even when the event is lost. Renderer never touches fs; keep the preload bridge minimal. **Tail state is per-session and persistent (redesigned 2026-07-13):** the byte offset and line remainder live in one `TranscriptTail` per session id that outlives the preview ‚Äî mounting a preview *arms* the watcher, unmounting only *disarms* it, so reopening reads just the delta; closing the session (`transcript:drop`) forgets it. The first batch of any from-zero read is sent `reset=true` (the renderer replaces its cache instead of appending), so truncations, re-created tails, and in-flight-delivery races can never duplicate items by construction.
- **Rendering:** user + assistant messages as markdown; **tool _noise_ is dropped, but the code Claude writes is rendered** (revised 2026-07-15). Tool activity was first collapsed to one-liners ‚Äî "‚öô Edit src/foo.ts" ‚Äî then (2026-07-14) dropped entirely: a working session's transcript is mostly tool_use/tool_result noise and the preview is the *human-readable* view. But "entirely" went too far ‚Äî the code Claude writes lives *only* in tool_use, and since *no* assistant turn interleaves prose with tool_use (verified empirically: in a real 3.7k-line transcript tool_use was ~74% of assistant items, and code turns are tool-only), dropping tool items silently dropped every code turn ‚Äî the preview of a coding session showed prose and no code. Carlos's call: **faint full-width line tint only** (code text keeps its normal color; the `+/-` markers stay the non-color signal), GitHub-diff style. The tint bleeds edge-to-edge and across horizontal scroll (`code` is `inline-block; min-width:100%`, lines are `border-box` blocks). Scoped to `diff` fences only ‚Äî every other code block, incl. full-file **Write** listings, falls through to marked's default (`return false`) and stays plain; coloring a whole new file green would just hurt readability. The override lives in a once-run module-script block, builds raw HTML so it escapes lines itself, and DOMPurify keeps the classes, language hinted from the file extension, the fence sized to outrun any backtick run inside. Every *other* tool (Read, Bash, Grep, tool_result‚Ä¶) still stays dropped as noise. (Known trade-off: a full-file Write dumps the whole file into the pane; accepted ‚Äî it's a read-only scrollable view ‚Äî cap later if it bites.) **Sanitize before injecting HTML** (conversation content is untrusted input ‚Äî unsanitized markdown‚ÜíHTML is self-inflicted XSS), and **forbid inline `style` on top of DOMPurify's defaults** (review 2026-07-13): with inline CSS a prompt-injected message could `position:fixed` an overlay over the app's own UI and spoof it.
